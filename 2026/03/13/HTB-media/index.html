HTB Media Machine

Created2026-03-13|Updated2026-04-17
|Post Views:

Media is a medium rated Windows machine hosted on HackTheBox. It starts with a rather unassuming web page, with little to be explored. The primary download functionality interacts with Windows Media Player, allowing us to upload a payload that steals the users NTLM hash. After cracking the encrypted password, we can maneuver around write permissions to gain access to the second user. The final step gives a choice of privileges to exploit in order to achieve authority/system.

Initial Enumeration

To begin, the nmap scan reveals only three ports open: SSH (22), HTTP (80), RDP (3389)

Using curl or upon getting a 404 response, we find the server and PHP version.

As we explore the site, it seems there is nothing significant except for the upload function.

Foothold on MEDIA

Immediately, I notice that it allows the user to upload any file type. However, the response message seems important.

After trying a few scf and XSS files, I moved on to begin researching windows media player exploits/vulnerabilities, as that seemed the intended path. Though various CVEs pop up from a google search, it seems rather inefficent to take shots in the dark without knowing the version. Upon further research, I found a “vulnerability“ that’s essentially built into Windows NTLM.

Upon uploading this payload, responder receives the hash of the enox user.

This hash can be cracked easily, giving us SSH credentials.

Maneuvering to Local Service

When we login, we fail to find any immediate routes towards privesc. The only interesting thing we find exists within the Documents folder, review.ps1.

Essentially, this powershell script checks the todo file periodically and opens the uploads with the media player. Weirdly, the Uploads folder is not in the same place as the webroot, which is typical of webapp challenges in HTB. Instead, it’s located at C:\xampp\htdocs, where index.php handles the post requests.

Here, we can see how the application hashes the given parameters and handles uploads. With the provided clues, it seems as though the escalation factor is related to these folders. Using icacls, we can see that everyone has access to C:\...\Uploads, while Administrators and Service Accounts control C:\xampp\htdocs. Initially, this made me believe symbolic links may be the creative solution. However, this is banned because we are not admin.

Upon using varying arguments of the mklink command, we see two other options and one gives promising results.

Junction links exists exclusively on NTFS filesystems and are capable of redirecting entire folders, given we have write access to the first. Keeping that in mind, we can use the previous information we gained on how the hash folders are created to get a webshell.

After removing the folder, let’s recreate the folder with mklink.

Then, upon uploading the payload…

We get a webshell as local service!

Using a base64 encoded payload from revshells.com, we now have remote access.

At this point, there are two privilege escalation methods, one exploiting potato attacks and the other, SeTcbPrivilege. Although SeImpersonate may be missing initially, it can be retreived as seen in this very interesting post.

However, given that a vulnerable privilege is already present, we’ll stick to the intended method. We can obtain a PoC from this github repo. Using Invoke-WebRequest, I’ll upload the escalation executable and my meterpreter payload. With that, we achieve authority\system.

If you’d like to try the other method, it’s actually quite simple using the provided PoC.

Now, we can use the latest and most convenient privilege escalation program, GodPotato. Just set up a listener and ta-da!

Beyond the Flag (Vulnerability Analysis)

“Trust me bro; it’s a feature” - Definitely Microsoft (WMP Vulnerability)

Over the years, Windows media player has been vulnerable to over 15 CVEs, many of them being RCE exploits. However, one “vulnerability” has remained unfixed, since Microsoft considers it a feature or the problem as expected behavior. Though it could be argued as NTLMs issue as a whole, Windows Media Player enables an attacker to leak NTLM hashes, using either UNC paths or even URLs. An attacker will use .asx, .wax, .wvx, or .wmx files to sneak in their own address, to which Windows will automatically attempt to authenticate in the case of using an UNC path. With URLs, it only authenticates immediately if the attacker server is misconfigured to be considered within the “Computer/Local Machine” Zone. Since this an intentional function on Microsofts part, there is no single patch or update to fix this. Nonetheless, security professionals could still refrain from using this software, implement e-mail filters for these unsuspecting extensions, or contain attachments within isolated VMs, ensuring callbacks are impossible.

Wait, this is actually just a feature (Directory Junctions)

Like the first, this issue that leads to lateral movement is also not a vulnerability, even more so than the last. Rather, it’s simply how Windows NTFS systems interacts with write permissions in regards to junctions. When an user with write access creates this folder in a directory, they may redirect the following input to another directory that they do not necessarily have access to. So, in application to this machine, when the enox user creates a junction, only the web app user needs the correct permissions to both directories in order to create the payload at the root directory. This is the reason why enox cannot edit the web root itself, but can create a junction that the file upload will travel through with local service rights. To prevent this from occuring, the admin should have been more nuanced in the amount of permissions given to each user. If you want to do further research on the intended uses of Windows file pointers, here’s a helpful resource.

Finally, actual vulnerabilities (Potatoes and CVE-2024-21304)

The privilege escalation step can be approached in two ways, either exploiting SeTcbPrivilege or SeImpersonatePrivilege. Upon initial enumeration, the only notable privilege via whoami /priv is SeTcbPrivilege. However, since we know this is a service account, it’s known that SeImpersonate should be present by default. To recover the privilege, FullPowers uses a task scheduler (in the service context) and token manipulation to bypass privilege restrictions. By doing this, we can obtain the missing SeImpersonatePrivilege, which is exploitable by one of the many potatoes, originally PrintSpoofer. At the moment, there is no way to completely mitigate this vulnerability, besides keeping Windows updated and removing the privilege entirely. Admins should carefully monitor any service accounts possessing this privilege and justify its necessity. If we don’t want to go this route, alternatively, we can exploit CVE-2024-21304 as well. The vulnerability comes from the lack of enforced access control in TCB operations, leading to invalidated user input. It affects versions 10.0 to 10.0.17763.5458, when it was patched.
